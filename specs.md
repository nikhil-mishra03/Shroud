# Design: Vault — Visible Security Co-Pilot for AI Tools

## Problem Statement

Developers want to point AI coding tools (Claude Code, Aider, etc.) at real data — production logs,
env files, customer records, internal APIs — but can't do it safely. The data contains secrets: API
keys, emails, IP addresses, tokens. Security teams block it. Engineers either avoid AI (losing
productivity) or use it unsafely (risking leaks).

The insight: this isn't a policy problem, it's a tooling problem. There's no transparent layer that
sits between the developer's workflow and the LLM without requiring any integration work.

---

## What Makes This Cool

Vault flips the security layer from **invisible** to **visible**. Instead of silently filtering data
in the background, it shows the developer exactly what it caught:

> "Secrets detected. This data would have been sent to Anthropic. Masking it now."

The visibility layer is the product. It builds trust, makes the protection feel real, and turns a
utility into something you'd screenshot and share. Developers who see what *would have* leaked —
without it actually leaking — immediately understand the value.

The optional split-pane TUI (activated via `vault run --ui`) shows a live session audit: left pane
is what you typed, right pane is what the LLM actually received. Masked tokens highlighted.
Session summary on exit.

---

## Architecture Decision: Go (not Python)

The proxy is a **high-performance, real-time network proxy + streaming processor**. Latency is a
first-class constraint — users must not feel a perceptible difference between running with and
without Vault.

Go is the right choice:
- Goroutines for concurrent request handling with near-zero overhead
- No GIL — true parallelism on streaming chunk processing
- Sub-millisecond regex substitution on the hot path
- Single binary distribution — `brew install` or `curl | sh`, no runtime deps

Python would be fine for a prototype but the latency would be noticeable on every LLM call.

---

## Constraints

- Latency overhead must be imperceptible (target: <5ms added latency on the proxy hop)
- Zero config required for the base case — `vault run claude` just works
- Local-only — no data leaves the machine, no cloud dependency
- Process-level wrapping — no modification to the AI tool, the user's shell, or their code
- MVP: Anthropic API only (OpenAI in phase 2)
- MVP: Regex/pattern masking only (NLP-based detection in phase 2)
- Visibility layer is opt-in — silent by default, `--ui` flag to enable

---

## Premises

1. **Streaming rehydration is the core technical risk** — masking outbound is straightforward with
   regex, but replacing `[EMAIL_1]` back in a streaming token-by-token response without corrupting
   the stream requires careful buffering. This is the thing most likely to make v1 feel broken.
   *(Agreed)*

2. **The visibility layer IS the product** — an invisible proxy is a nice utility; a visible
   security co-pilot is something you open a PR about. *(Agreed)*

3. **Intelligent policy engine is phase 2** — shipping it in v1 adds scope (persistence, learning,
   rule management) and risks delaying the thing that matters. *(Agreed)*

4. **Anthropic API only for MVP** — you use Claude Code, ship what you use. *(Agreed)*

5. **Session-scoped logging now, daemon as phase 2** — logs live for the duration of `vault run`,
   written to `~/.vault/sessions/<timestamp>.jsonl` on exit. Design the format now so the daemon
   upgrade is clean. *(Agreed)*

## Recommended Approach

**Approach B — Go proxy + optional TUI**, with the following design:

### CLI UX
```
vault run claude           # silent proxy, zero noise
vault run --ui claude      # proxy + split-pane TUI
vault status               # show if proxy is running, current session stats
vault logs                 # show last session's masked token summary
```

### Process Architecture
```
vault run [--ui] <tool>
    │
    ├── spawns: local HTTP proxy (localhost:randomport)
    │     ├── injects ANTHROPIC_BASE_URL into child env
    │     ├── intercepts outbound requests → mask → forward to Anthropic
    │     └── intercepts inbound responses → rehydrate → return to tool
    │
    ├── spawns: <tool> as child process (e.g. claude)
    │
    ├── writes: ~/.vault/sessions/<timestamp>.jsonl (live, flushed per event)
    │
    └── [--ui only] spawns: TUI process reading from session log via unix socket
```

### Masking Engine (MVP)
Rule-based regex patterns, deterministic within session:

| Entity   | Pattern                        | Placeholder  |
|----------|--------------------------------|--------------|
| Email    | RFC 5322 regex                 | `[EMAIL_1]`  |
| IP addr  | IPv4 + IPv6                    | `[IP_1]`     |
| API key  | `sk-...`, `Bearer ...`, etc.   | `[KEY_1]`    |
| Token    | JWT pattern, hex 32+ chars     | `[TOKEN_1]`  |
| Env vars | `KEY=value` in context         | `[ENV_1]`    |

Counter increments per entity type per session. Mapping stored in-memory, written to session log.

### Streaming Rehydration
- Buffer incoming stream chunks
- Scan buffer for placeholder tokens (`[TYPE_N]` pattern)
- If placeholder spans chunk boundary, hold partial match until next chunk arrives
- Replace in-place, flush to tool's stdout
- Worst case: one extra chunk of buffering latency (~ms)

### Session Log Format (`~/.vault/sessions/<timestamp>.jsonl`)
```jsonl
{"type":"session_start","ts":"...","tool":"claude","pid":12345}
{"type":"mask_event","ts":"...","entity":"EMAIL","placeholder":"[EMAIL_1]","request_id":"abc"}
{"type":"mask_event","ts":"...","entity":"KEY","placeholder":"[KEY_1]","request_id":"abc"}
{"type":"rehydrate_event","ts":"...","placeholder":"[EMAIL_1]","request_id":"abc"}
{"type":"session_end","ts":"...","masked_count":4,"request_count":12}
```

No real values are ever written to the log. Only entity type + placeholder.

### TUI Layout (`--ui` mode)
```
┌─ Vault ──────────────────────────────────────────────────────────────┐
│ Session: 2026-03-22 18:05  │  Tool: claude  │  Masked: 3 secrets    │
├──────────────────────┬───────────────────────────────────────────────┤
│  YOU TYPED           │  LLM RECEIVED                                  │
│                      │                                                │
│  john@company.com    │  [EMAIL_1]                    ← masked        │
│  accessed 10.2.3.4   │  accessed [IP_1]              ← masked        │
│  with sk-abc123      │  with [KEY_1]                 ← masked        │
│                      │                                                │
│  [response]          │  Investigate [IP_1]                            │
│  Investigate         │  → rehydrated: 10.2.3.4       ← rehydrated   │
│  10.2.3.4            │                                                │
└──────────────────────┴───────────────────────────────────────────────┘
  ↑ secrets blocked this session            q: quit TUI (proxy keeps running)
```

## Phase 2 Roadmap (not in scope now, but design for it)

1. **Intelligent policy engine** — if entity type X appears in >3 requests, auto-create a
   persistent rule. `vault policies` to view/edit. Stored in `~/.vault/policies.yaml`.
2. **Daemon mode** — persistent background process, cross-session history, `vault history`
3. **NLP-based detection** — replace regex with a local model (e.g. spaCy or a small ONNX model)
   for semantic PII detection (names, company names, addresses)
4. **OpenAI API support** — same proxy, different base URL injection
5. **`vault ui`** — local web dashboard (the Approach C upgrade)

---

## Open Questions

1. **TUI library choice** — [bubbletea](https://github.com/charmbracelet/bubbletea) (Go, idiomatic)
   vs. writing a minimal raw terminal renderer. Bubbletea is the obvious answer but adds a dep.
2. **Proxy port** — random ephemeral port (avoids conflicts) vs. fixed port (simpler config).
   Leaning random, injected into child env at startup.
3. **Partial token rehydration** — if the LLM responds with `[EMAIL` in one chunk and `_1]` in
   the next, the buffer logic needs to be airtight. Worth a dedicated test suite.
4. **Signal handling** — what happens when the child process (`claude`) exits vs. crashes vs.
   the user hits Ctrl+C? Proxy should clean up cleanly in all three cases.

---

## Success Criteria

- `vault run claude` adds <5ms latency on the proxy hop (measurable with `time`)
- All MVP entity types (email, IP, key, token) are masked 100% of the time in a test suite
- Streaming rehydration works correctly across chunk boundaries (no broken placeholders in output)
- Session log is written on exit and readable via `vault logs`
- `vault run --ui claude` renders the split-pane TUI without corrupting the child process I/O
- Single binary, installable with one command, no runtime dependencies

---

## The Build Order

1. **Proxy core** — Go HTTP proxy, env injection, child process spawning, clean signal handling
2. **Masking engine** — regex patterns for MVP entity types, in-memory session mapping
3. **Streaming rehydration** — buffered chunk scanner with partial-match handling
4. **Session logging** — JSONL writer, `vault logs` reader
5. **TUI** (`--ui` flag) — bubbletea split-pane, reads from session log via unix socket
6. **CLI polish** — `vault status`, colored output, `vault logs` summary

Each step is independently testable and shippable. Don't start step 2 until step 1 works.
